# Phase 1: Core Infrastructure — Re-Audit
*Previous verdict: **PASS WITH WARNINGS***

## Changes Since Last Audit

### F-5 — OTP Brute-Force Lockout ✅ FIXED
**Previous:** `OTPVerifyView` had no attempt counter. A 6-digit OTP was brute-forceable in 10 minutes.

**Current state:**
```python
_OTP_MAX_ATTEMPTS = 5

# In OTPVerifyView.post():
attempts = request.session.get("otp_attempts", 0)
if attempts >= _OTP_MAX_ATTEMPTS:
    OTPCode.objects.filter(user_id=user_id, is_used=False).update(is_used=True)
    request.session.flush()
    # → redirect to login with error
```
After 5 wrong codes the OTP is invalidated and the session is flushed — the attacker must restart the login flow and trigger a new OTP. Attempt counter resets on `OTPResendView` (new code issued) and on successful verify.

**Remaining concern:** The lockout is session-scoped, not IP-scoped. A sophisticated attacker with 5 browser sessions could make 25 attempts. Recommend `django-axes` for IP-level rate limiting in a production hardening sprint.

### F-6 — API Throttling ✅ FIXED
**Previous:** `DEFAULT_THROTTLE_CLASSES` absent from DRF settings.

**Current state:** `settings/base.py` now contains:
```python
"DEFAULT_THROTTLE_CLASSES": [
    "rest_framework.throttling.AnonRateThrottle",
    "rest_framework.throttling.UserRateThrottle",
],
"DEFAULT_THROTTLE_RATES": {"anon": "60/minute", "user": "300/minute"},
```
All API views inherit these limits. Auth endpoints should eventually get their own tighter scope (e.g., `ScopedRateThrottle` at `10/minute` for OTP).

### F-2 / F-3 — Default FiscalYear on Registration ✅ FIXED
**Previous:** New organizations had no FiscalYear. `_assert_period_open()` silently skipped all period checks, allowing backdated postings to any date.

**Current state:** `seed_default_fiscal_year(org)` is called from `RegisterView.save()` alongside COA and tax-code seeding:
- Creates `FiscalYear(name="FY {year}", start=Jan 1, end=Dec 31, status=OPEN)`
- Calls `GenerateFiscalPeriods` → 12 `AccountingPeriod` rows (OPEN)
- Idempotent (no-op if FiscalYear already exists)

**Remaining concern:** Only the current calendar year is created. If a new tenant wants to post opening balances for a prior year, they must manually create that FiscalYear.

## Remaining Open Items (unchanged)
- OTP resend reuses existing code if >60 s remaining (minor UX issue, not a security risk)
- REST API lacks versioning (`/api/v1/`)
- `Subscription` billing enforcement absent
- Branch-scoped tenancy not enforced
- Incomplete audit log (login events, role changes not recorded)

## Verdict
**PASS WITH WARNINGS** → **PASS WITH WARNINGS** (unchanged rating, but two critical items closed)

The OTP lockout and API throttling close the two most urgent security gaps. FiscalYear auto-creation removes the backdated-posting risk for new tenants. Remaining warnings are lower priority.

# Phase 2: Core Accounting — Re-Audit
*Previous verdict: **PASS WITH WARNINGS***

## Changes Since Last Audit

### F-4 — `fiscal_period` FK Now Populated ✅ FIXED
**Previous:** `PostJournalEntry` never set `fiscal_period_id` on the created entry; the FK existed but was always NULL.

**Current state:**
```python
fiscal_period_id = _find_open_period_id(draft.entry_date)
entry = JournalEntry(
    ...
    fiscal_period_id=fiscal_period_id,
)
```
`_find_open_period_id()` queries `AccountingPeriod` by year+month matching `entry_date` and returns the PK of the first open period (or `None` if no period exists). Every posted entry now carries an explicit period link, enabling period-level reporting without date-range queries.

**Remaining concern:** If an org has no AccountingPeriods yet (e.g., created before F-2 was deployed), `fiscal_period_id` remains NULL. The fix is non-breaking: old entries stay NULL, new entries get the FK.

### F-7 — `entry_number` Unique Per Org (DB-Level) ✅ FIXED
**Previous:** `entry_number` was assigned via a post-save `UPDATE`. A race condition between two concurrent posts could generate duplicate numbers before the UPDATE fired. No DB constraint prevented this.

**Current state — `finance/infrastructure/models.py`:**
```python
models.UniqueConstraint(
    fields=("organization", "entry_number"),
    condition=models.Q(entry_number__gt=""),
    name="finance_journal_entry_unique_entry_number_per_org",
),
```
Partial constraint (non-empty strings only) allows the blank default during creation and enforces uniqueness once `entry_number` is set. Migration `0011_qa_fixes.py` applied successfully.

**Note:** MySQL does not enforce partial `UniqueConstraint` with conditions (W036 warning). On MySQL the race condition risk remains at DB level but is still guarded by the post-save UPDATE logic. Recommend migrating to PostgreSQL or adding an explicit DB-level trigger for full enforcement.

### F-11 — `Account.level` Cascade on Parent Move ✅ FIXED
**Previous:** Moving an account under a new parent correctly updated its own `level`, but all descendants kept stale levels.

**Current state — `Account.save()`:**
```python
if self.pk:
    old_parent_id = Account.objects.filter(pk=self.pk).values_list(...).first()
    parent_changed = old_parent_id != self.parent_id
...
super().save(*args, **kwargs)
if parent_changed:
    self._update_descendants_level()
```
`_update_descendants_level()` walks direct children and calls `.save()` on each, triggering their own cascade. Works for arbitrary tree depth.

**Performance note:** Deep COA trees with many children will trigger N save() calls. This is acceptable for a typical COA (rarely >3 levels deep, moves are rare admin operations).

## Remaining Open Items (unchanged)
- `ApproveJournalEntry` use case absent (draft → approved step unreachable)
- `PostJournalEntry` bypasses status workflow (creates POSTED directly)
- No foreign-currency revaluation use case
- `ClosingRun` + `AdjustmentEntry` UI absent
- No running-balance cache (scale risk at 100k+ lines)

## Verdict
**PASS WITH WARNINGS** → **PASS WITH WARNINGS** (unchanged rating, three gaps closed)

`fiscal_period` FK now populated, `entry_number` uniqueness enforced at model level, level cascade corrected. The `ApproveJournalEntry` workflow gap and status bypass remain — acceptable for MVP but should be addressed before any multi-user approval workflow is enabled.

## Changes Since Last Audit

### F-12 — `TaxTransaction` Created by `PostSale` ✅ FIXED (best-effort)
**Previous:** POS-style `PostSale` created no `TaxTransaction` rows — VAT audit trail was incomplete for all sales not going through `IssueSalesInvoice`.

**Current state — `PostSale._record_tax_transactions()`:**
```python
for line in draft.lines:
    if line.tax_rate_percent == Decimal("0"):
        continue
    tc = TaxCode.objects.filter(rate=line.tax_rate_percent, is_active=True).first()
    if tc is None:
        continue  # silent skip — GL is already correct
    CalculateTax().execute(CalculateTaxCommand(
        direction=TaxDirection.OUTPUT,
        source_type="sales.sale",
        source_id=sale_id,
        journal_entry_id=je_id,
        ...
    ))
```
Lines with a non-zero rate now produce a `TaxTransaction` if a matching active `TaxCode` exists. Wrapped in `try/except` so a missing TaxCode never aborts the sale posting (non-fatal).

**Known limitation:** The match is by rate value (e.g., 15.00), not by TaxCode ID. If two TaxCodes share the same rate, the first one found is used. This is acceptable for the standard 3-code setup (VAT15, VAT0, VATEX) but would need a line-level `tax_code_id` in `SaleLineSpec` for full accuracy. Tracked as a future improvement.

# Phase 6: Tax & Period Closing — Re-Audit
*Previous verdict: **FAIL***

## Changes Since Last Audit

### F-1 — Input VAT Account Wired on Seeded TaxCode ✅ FIXED (CRITICAL)
**Previous failure:** `seed_default_tax_codes()` created `VAT{rate}%` with `input_tax_account_id = None`. Every purchase using this code failed to post input VAT to a recoverable-asset account — balance sheet understated assets, P&L overstated expenses.

**Current state:**
```python
tax_recoverable = Account.objects.filter(organization=organization, code="1600").first()

TaxCode.objects.create(
    ...
    output_tax_account_id=vat_payable.pk if vat_payable else None,
    input_tax_account_id=tax_recoverable.pk if tax_recoverable else None,  # ← FIXED
)
```
Account 1600 (Tax Recoverable, asset type) is now seeded as part of the COA and wired as `input_tax_account` on the standard VAT code. `IssuePurchaseInvoice` uses `output_tax_account` for sales and `input_tax_account` for purchases — both are now correctly linked.

**Edge case confirmed safe:** `tax_recoverable.pk if tax_recoverable else None` — if account 1600 was not created (unlikely, but possible on pre-fix orgs), the code degrades gracefully instead of crashing. Existing organizations that registered before this fix will still have `input_tax_account = None`; a data migration or admin action is needed for them.

### F-12 — `TaxTransaction` for POS Sales ✅ PARTIALLY FIXED
See Phase 3 re-audit. POS sales now attempt to create `TaxTransaction(direction=OUTPUT)` rows when a matching `TaxCode` exists.

## Remaining Open Items

### Still Missing (carry-over from prior audit)
- **VAT Return / VAT settlement use case** — no use case that nets output vs. input `TaxTransaction` and creates a VAT payment entry. This is the highest-priority remaining gap for legal compliance.
- **`ClosingRun` + `AdjustmentEntry` + `PeriodSignOff` UI** — models and use cases exist; no web views.
- **`TaxProfile` orphan model** — still not wired to any document flow.
- **`CalculateTax` REST endpoint** — not exposed via API.
- **Input VAT reconciliation** — no use case verifies `TaxTransaction(input)` amounts match the `tax_recoverable_account` GL balance.

### Scope of Fix for Existing Orgs
Organizations that registered **before** this fix were seeded with `input_tax_account = None`. They will need one of:
1. A management command that back-fills `input_tax_account_id` on existing TaxCode records.
2. Manual update via the TaxCode admin interface.

## Verdict: FAIL → **PASS WITH WARNINGS**

**The critical blocker (input VAT account = None) is resolved.** New organizations registered after this fix will have correct input VAT accounting from day one. The phase moves from FAIL to PASS WITH WARNINGS because the legal compliance minimum (correct GL posting for input VAT) is now met, while the VAT Return report and period-close UI remain outstanding.

# Phase 7: Intelligence & Reporting — Re-Audit
*Previous verdict: **FAIL***

## Changes Since Last Audit

### F-9 — `_org()` None Guard → 403 ✅ FIXED
**Previous failure:** When `_org(request)` returned `None` (no TenantContext, no OrganizationMember), all intelligence queries received `organization_id=None`. This produced either a Django crash (when passed to `get_object_or_404`) or silently returned all records from all tenants (when passed to `.filter()`).

**Current state — `_require_org(request)`:**
```python
def _require_org(request):
    org_id = _org(request)
    if org_id is None:
        raise PermissionDenied("No active organization found for this user.")
    return org_id
```
All 25 intelligence views now call `_require_org(request)` instead of `_org(request)`. Any unauthenticated-but-somehow-authenticated call without an org returns **403**, not a crash or data leak.

### F-10 — `IsFinanceManager` on Action Views ✅ FIXED
**Previous failure:** Dismiss-anomaly, resolve-anomaly, review-duplicate, assign/transition-audit-case, create/update alert-rules, and acknowledge alert-events were all accessible by any authenticated organization member regardless of role.

**Current state — 8 views hardened:**
```
AnomalyCaseAssignView    → [IsAuthenticated, IsFinanceManager]
AnomalyCaseResolveView   → [IsAuthenticated, IsFinanceManager]
DuplicateMatchReviewView → [IsAuthenticated, IsFinanceManager]
AuditCaseAssignView      → [IsAuthenticated, IsFinanceManager]
AuditCaseTransitionView  → [IsAuthenticated, IsFinanceManager]
AlertRuleListView (POST) → [IsAuthenticated, IsFinanceManager]
AlertRuleDetailView (PUT)→ [IsAuthenticated, IsFinanceManager]
AlertEventAcknowledgeView→ [IsAuthenticated, IsFinanceManager]
```
Read-only views (list, detail) remain `IsAuthenticated` — any org member can view anomalies and KPIs. Only managers can act on them.

## Remaining Open Items (major, unchanged)

### Not Implemented (scaffolding only)
- **`DetectAnomalies`** use case — `AnomalyCase` records cannot be created automatically; the model exists but no computation populates it.
- **`ComputeRiskScores`** use case — `RiskScore` model exists; no score is ever calculated.
- **`DetectDuplicates`** use case — `DuplicateMatch` model exists; no matcher runs.
- **`AlertRule` evaluation** — `AlertRule.condition_json` is stored but never evaluated. No Celery task fires `AlertEvent` records.
- **Celery tasks** — the entire intelligence async layer (periodic KPI snapshots, anomaly scanning, alert evaluation) has no `tasks.py`.
- **`AssistantQuery` LLM call** — endpoint exists; actual model call implementation unverified (may be a stub).

### Missing Financial Reports
- **Balance Sheet** — not implemented.
- **Income Statement (P&L)** — formatted statement absent (KPI `gross_margin` exists but is not a report).
- **Cash Flow Statement** — not implemented.

### Still Acceptable
- `AlertRule.condition_json` uses a structured JSON field (not `eval`) — **no code injection risk** (confirmed in prior audit, unchanged).
- `AssistantQuery` data exfiltration concern remains open until the LLM integration is implemented and reviewed.

## Verdict: FAIL → **FAIL** (unchanged)

The two critical security fixes (F-9, F-10) are meaningful improvements: intelligence views no longer crash on missing org context, and fraud-alert dismissals now require manager role. However, the underlying reason for the FAIL verdict — that the intelligence computation layer (anomaly detection, risk scores, alert evaluation, Celery tasks) is unimplemented scaffolding — is unchanged. The three statutory financial reports also remain absent.

**The phase remains FAIL but is materially safer than before.** It should be re-rated PASS WITH WARNINGS only after at least one of the following is implemented: anomaly detection, alert evaluation, or a full financial report (Balance Sheet or Income Statement).
